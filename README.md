# Контроль СИЗ и оборудования

Веб-приложение для учёта и контроля средств индивидуальной защиты (СИЗ),
материалов и оборудования по подразделениям, складам и персоналу сетевой
организации.

Система ведёт учёт имущества, отслеживает **сроки эксплуатации** и **сроки
поверки**, фиксирует фактическое местонахождение (на складе или у сотрудника)
и показывает, что находится в сроке, что скоро выходит из срока и что уже
просрочено.

## Возможности

- **Dashboard** — сводка: всего позиций, на складе, у сотрудников, в сроке,
  скоро истекает (30 дней), просрочено, поверка истекает/просрочена; графики и
  разбивка по подразделениям. Фильтрация по подразделению для РЭС.
- **Справочники** — категории, подкатегории и номенклатура СИЗ и
  материалов/оборудования; редактируются через интерфейс.
- **Реестр персонала** — сотрудники с привязкой к подразделению.
- **Склад** — позиции на складах, сгруппированные по складам подразделений.
- **Учёт единиц** — инвентарные/серийные номера, сроки, статусы, история.
- **Операции** — выдача сотруднику, возврат на склад, перемещение между
  подразделениями, регистрация поверки.
- **Контроль сроков** — отдельные представления: истекает эксплуатация,
  просрочено по эксплуатации, истекает поверка, просрочена поверка.
- **Журнал движения** — неизменяемая история операций.
- **Отчёты** — остатки, выданное, по сотруднику, просроченные, истекающие,
  поверка, движение; выгрузка в **Excel** и **CSV**.
- **Авторизация** — вход по логину/паролю, JWT, пароли хранятся в виде bcrypt-хэшей.
- **Права доступа** проверяются на стороне backend, а не только в интерфейсе.

## Архитектура

**Backend** построен слоями:

- `models/` — таблицы БД (SQLAlchemy ORM).
- `schemas/` — контракты API (Pydantic): отдельные схемы на ввод и вывод.
- `routers/` — HTTP-эндпоинты, сгруппированные по доменам (`auth`, `users`,
  `employees`, `catalog`, `inventory`, `operations`, `reports`, `dashboard`,
  `journal`, `departments`).
- `services/` — бизнес-логика, не зависящая от HTTP:
  - `status.py` — расчёт сроков эксплуатации и поверки;
  - `reports.py` — построение отчётов и выгрузка в Excel/CSV;
  - `audit.py` — запись в журналы движения и аудита.
- `dependencies.py` — извлечение текущего пользователя из JWT и проверки прав
  (`require_admin`, `require_privileged`, ограничение по подразделению для РЭС).

Все эндпоинты под префиксом `/api`. Документация Swagger доступна на `/docs`.

**Frontend** — SPA на React. Токен хранится в `localStorage`, добавляется в
каждый запрос интерсептором axios. Маршруты защищены: при отсутствии токена —
редирект на `/login`; страницы вне прав роли скрыты в меню и закрыты на роутере.

**Расчёт статусов сроков** выполняется на лету при чтении (в `services/status.py`),
а не хранится в БД — это исключает необходимость в ежедневном фоновом задании
для пересчёта «просрочено/истекает».

## Роли и права

| Роль (`code`)         | Видимость                  | Права                                                            |
|-----------------------|----------------------------|-----------------------------------------------------------------|
| Администратор (`admin`)| Все подразделения          | Полный доступ, управление пользователями, журналы               |
| Лаборатория (`lab`)    | Все подразделения          | Учёт, операции, перемещение, поверка, справочники, отчёты        |
| Служба учёта (`sue`)   | Все подразделения          | То же, что Лаборатория                                           |
| Пользователь РЭС (`res_user`) | Только своё подразделение | Просмотр своих данных; без удаления, перемещения и поверки |

Права проверяются на backend. Пользователь РЭС не получит доступ к чужому
подразделению даже прямым API-запросом (ограничение применяется в зависимостях
и в запросах к БД).

## Бизнес-логика сроков

- Пока позиция **на складе**, срок эксплуатации **не идёт** (если для позиции
  не включён флаг «срок идёт на складе»).
- При **выдаче сотруднику** дата начала эксплуатации = дате выдачи; дата
  окончания = дата начала + нормативный срок (из позиции или из справочника).
- При **возврате** история сохраняется (видно, кто и когда держал позицию).
  Повторная выдача **не сбрасывает** дату начала эксплуатации.
- **Поверка** считается независимо от срока эксплуатации.
- Статусы: `В сроке` (> 30 дней), `Скоро истекает` (≤ 30 дней),
  `Просрочено` (дата в прошлом), `Не требуется` (поверка не предусмотрена),
  `Не начата` (на складе, эксплуатация не стартовала).
- Удаление — «мягкое» (флаг `is_active`), данные и история не теряются.

## Быстрый старт (локально)

Нужны **Python 3.11+** и **Node.js 18+**.

### 1. Backend

```bash
cd backend
python -m venv .venv
source .venv/bin/activate            # Windows: .venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env                 # при необходимости отредактируйте
python -m app.seed                   # создаёт таблицы, роли, подразделения, админа, демо-данные
uvicorn app.main:app --reload        # http://localhost:8000  (Swagger: /docs)
```

По умолчанию используется SQLite-файл `siz_control.db` — никакой внешней БД не
требуется.

### 2. Frontend

В отдельном терминале:

```bash
cd frontend
npm install
cp .env.example .env                 # VITE_API_URL можно оставить пустым локально
npm run dev                          # http://localhost:5173
```

Vite проксирует запросы `/api` на backend (`localhost:8000`), поэтому CORS
локально не мешает. Откройте http://localhost:5173 и войдите.

## Переменные окружения

### Backend (`backend/.env`)

| Переменная                    | Назначение                                            | По умолчанию           |
|-------------------------------|-------------------------------------------------------|------------------------|
| `DATABASE_URL`                | Строка подключения к БД                               | `sqlite:///./siz_control.db` |
| `SECRET_KEY`                  | Ключ подписи JWT (задайте длинную случайную строку)   | заглушка               |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Время жизни токена                                    | `720`                  |
| `ADMIN_LOGIN`                 | Логин первого администратора                          | `admin`                |
| `ADMIN_PASSWORD`              | Пароль первого администратора                         | `admin123`             |
| `ADMIN_FULL_NAME`             | ФИО администратора                                    | `Администратор системы`|
| `CORS_ORIGINS`                | Разрешённые источники фронтенда (через запятую или `*`)| `*`                    |
| `UPLOAD_DIR`                  | Каталог для вложений                                  | `./uploads`            |

Генерация `SECRET_KEY`:
```bash
python -c "import secrets; print(secrets.token_urlsafe(48))"
```

### Frontend (`frontend/.env`)

| Переменная     | Назначение                                  | По умолчанию |
|----------------|---------------------------------------------|--------------|
| `VITE_API_URL` | Базовый URL backend (пусто = прокси Vite)   | пусто        |

`DATABASE_URL` поддерживает форматы `postgres://`, `postgresql://` и
`postgresql+psycopg://` — приложение само приводит их к нужному виду.

## PostgreSQL

Объём данных небольшой, бесплатного тарифа достаточно.

### Вариант А. Supabase (Free)

1. Создайте проект на https://supabase.com.
2. Project Settings → Database → Connection string → **URI**.
3. Подставьте строку в `DATABASE_URL`, например:
   ```
   DATABASE_URL=postgresql://postgres:ВАШ_ПАРОЛЬ@db.xxxx.supabase.co:5432/postgres
   ```
4. Выполните миграции и seed (см. ниже).

### Вариант Б. Локальный PostgreSQL

```bash
createdb siz_control
export DATABASE_URL="postgresql://postgres:postgres@localhost:5432/siz_control"
```

## Миграции (Alembic)

Для удобства локального запуска приложение **создаёт таблицы автоматически**
при старте (`Base.metadata.create_all`). Для контролируемой эволюции схемы
используйте Alembic.

В комплекте уже есть начальная миграция (`alembic/versions/..._initial_schema.py`).

```bash
cd backend
alembic upgrade head          # применить миграции к БД из DATABASE_URL

# при изменении моделей:
alembic revision --autogenerate -m "описание изменения"
alembic upgrade head
```

После применения схемы наполните справочные/стартовые данные:
```bash
python -m app.seed
```

## Деплой на Render

В корне есть `render.yaml` (Blueprint), создающий три ресурса: PostgreSQL (free),
backend Web Service и frontend Static Site.

1. Залейте репозиторий на GitHub.
2. В Render: **New → Blueprint**, выберите репозиторий — Render прочитает
   `render.yaml`.
3. При создании задайте секреты (помечены `sync: false`):
   - `ADMIN_PASSWORD` — пароль администратора;
   - `CORS_ORIGINS` — пока можно `*`, затем уточнить;
   - `VITE_API_URL` — пока можно оставить пустым.
4. После первого деплоя у вас будут два URL, например:
   - backend: `https://siz-control-api.onrender.com`
   - frontend: `https://siz-control-web.onrender.com`
5. Пропишите перекрёстные значения и передеплойте:
   - в backend `CORS_ORIGINS` = URL фронтенда;
   - в frontend `VITE_API_URL` = URL backend (Static Site читает переменную на
     этапе сборки, поэтому нужен повторный деплой фронтенда).

Backend запускается командой
`python -m app.seed && uvicorn app.main:app --host 0.0.0.0 --port $PORT` —
`seed` идемпотентен: создаёт роли/подразделения/администратора один раз и
добавляет демо-данные только в пустую базу.

`DATABASE_URL` и `SECRET_KEY` Render подставляет автоматически (БД — из
блока `databases`, ключ — `generateValue: true`).

> Бесплатный Web Service на Render «засыпает» при простое — первый запрос после
> паузы может занять 30–60 секунд.

### Деплой без Blueprint (вручную)

- **Backend** → New Web Service, root `backend`,
  build `pip install -r requirements.txt`,
  start `python -m app.seed && uvicorn app.main:app --host 0.0.0.0 --port $PORT`.
  Переменные: `DATABASE_URL`, `SECRET_KEY`, `ADMIN_LOGIN`, `ADMIN_PASSWORD`, `CORS_ORIGINS`.
- **Frontend** → New Static Site, root `frontend`,
  build `npm install && npm run build`, publish `dist`,
  переменная `VITE_API_URL`. Добавьте rewrite-правило `/*` → `/index.html`.
- **Database** → New PostgreSQL (free), строку подключения — в `DATABASE_URL`.

## Как пользоваться

1. **Войдите** под нужной учётной записью.
2. **Справочники** → заведите категории/подкатегории и номенклатуру СИЗ и
   оборудования (нормативный срок, необходимость и периодичность поверки).
3. **Персонал** → добавьте сотрудников подразделения.
4. **СИЗ / Материалы и оборудование** → добавьте единицы учёта (инв./серийный
   номер, подразделение-владелец, склад, дата поступления). Пока позиция на
   складе — срок эксплуатации не идёт.
5. **Выдача** → выберите сотрудника и позиции; с даты выдачи стартует срок
   эксплуатации.
6. **Возврат** → примите позиции от сотрудника, указав состояние (исправно /
   требует проверки / требует списания / утеряно).
7. **Перемещение** → переместите позицию между подразделениями/складами
   (Лаборатория, Служба учёта, Администратор).
8. **Поверка** → зарегистрируйте поверку и продлите срок следующей поверки.
9. **Контроль сроков** → отслеживайте истекающие и просроченные сроки.
10. **Отчёты** → сформируйте отчёт и выгрузите в Excel/CSV.
11. **Журнал действий** → просматривайте историю операций.
12. **Пользователи** (только администратор) → создавайте учётки, назначайте роли
    и подразделения, блокируйте доступ.

## Принятые упрощения

Реализация следует ТЗ; ради чистоты и работоспособности приняты решения:

- **Фиксированные наборы значений** (тип имущества, статусы, единицы срока,
  роли) реализованы перечислениями в коде, а **категории и подкатегории** — как
  редактируемые таблицы (как требует ТЗ).
- **Статусы сроков** вычисляются при чтении, а не хранятся в БД — не нужен
  фоновый пересчёт.
- **SQLite по умолчанию** для нулевой настройки локально; в продакшене —
  PostgreSQL через `DATABASE_URL`. Код одинаков для обеих СУБД.
- **Вложения файлов**: таблица `files` и связи предусмотрены в модели, но из-за
  эфемерной файловой системы Render загрузка файлов в комплекте не активирована.
  Для постоянного хранения подключите внешнее хранилище (Supabase Storage / S3).
- **Удаление — мягкое** (`is_active`), критичные операции пишутся в журналы.
